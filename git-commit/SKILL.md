# Git 提交技能

## 何时使用

- 用户输入 `/git-commit`
- 用户说"提交代码"、"commit"、"提交一下"
- 用户想提交当前代码变更

## 执行流程

**严格按以下顺序执行：**

### 第一步 — 暂存所有变更

先执行 `git status` 检查是否有敏感文件，再执行 `git add .` 暂存所有变更。

**敏感文件检测** — 以下文件不应被自动提交，发现后必须先警告用户：

```
- .env, .env.*, *.key, *.pem, credentials*, secrets*
- 用户明确排除的文件
```

- 如果检测到敏感文件 → **暂停，警告用户**，等待确认后再继续
- 如果没有任何变更（工作区干净）→ **直接告知用户，不创建空提交**

### 第二步 — 收集上下文

并行执行以下操作：

1. `git status` — 列出所有变更文件
2. `git diff --cached --stat` — 暂存区变更统计
3. `git log --oneline -10` — 最近 10 条提交记录（学习前缀风格和分隔符）
4. 读取项目根目录的 `CLAUDE.md`（如存在）— 获取项目级提交约束

### 第三步 — 分析变更内容

逐文件读取 diff（`git diff --cached -- <路径>`），理解每处变更的具体意图。按模块/页面分组。

**如果无法理解某处变更的意图，必须先问用户，不要猜测。**

### 第四步 — 生成 Commit Message

按以下三段式格式生成 commit message：

```
<类型><分隔符><一行概述>

变更文件：
  <树形结构展示所有变更文件>

变更概览：
  - <模块/页面>：<做了什么>
    - <具体变更描述>
    - <具体变更描述>
  - <模块/页面>：<做了什么>
    - <具体变更描述>
```

#### 类型判定规则

| 条件                       | 类型       |
| -------------------------- | ---------- |
| 新增功能 / 页面 / 组件     | `feat`     |
| 修复 Bug                   | `fix`      |
| 重构 / 代码结构优化        | `refactor` |
| 样式 / CSS 调整            | `style`    |
| 文档 / 注释变更            | `docs`     |
| 构建 / 工具 / 配置 / 依赖  | `chore`    |
| 测试相关                   | `test`     |
| 性能优化                   | `perf`     |

- **多类型混合时**：选择最主要的那一个；如果真的分不清主次，问用户
- **分隔符**：从最近的 git log 中学习。常见模式：
  - `feat：xxx` → 全角冒号 `：`
  - `feat: xxx` → 半角冒号+空格 `: `
  - `feat:xxx` → 半角冒号无空格
  - 如果历史记录中没有参考，默认用 `：`
- **语言**：匹配最近提交记录使用的语言（中文、英文等）

#### 文件树格式

使用 `├──` 和 `└──` 连接符构建树形结构。示例：

```
变更文件：
  src/app/pages/Login/
  ├── index.tsx
  ├── components/
  │   └── LoginForm.tsx
  └── hooks/
      └── useAuth.ts
  src/app/api/auth/
  └── index.ts
```

### 第五步 — 执行提交

直接执行：

```bash
git commit -m "<用户确认的 commit message>"
```

### 第六步 — 询问是否推送

提交成功后，使用 `AskUserQuestion` 工具询问用户是否要 push 到远程仓库，提供"推送"和"暂不推送"两个选项。用户确认推送后再执行 `git push`。

## 铁律

1. **禁止水印** — 绝不添加 "Co-Authored-By: Claude"、"Generated with Claude Code" 等任何水印
2. **不懂就问** — 变更意图不明确时，问用户，不要猜
3. **不自动推送** — 只 commit，push 必须用户明确要求
4. **尊重项目规范** — CLAUDE.md 和 git log 中的约定优先于默认规则
5. **禁止空提交** — 没有变更就告知用户并停止
6. **敏感文件拦截** — .env、密钥、凭证等文件必须先警告再决定是否提交
7. **禁止 amend** — 始终创建新提交，绝不 amend
8. **提交后必问推送** — 每次提交成功后，都必须询问用户是否要 push
